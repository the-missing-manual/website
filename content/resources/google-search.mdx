---
title: Searching Effectively
description: How to get better results out of search engines and find what you actually need faster.
---

The quality of what you find online depends almost entirely on how you search. Most people type a vague phrase and scroll through results. Learning to search precisely changes what's available to you.

### Operators That Actually Help

These work on most major search engines.

**Exact phrase.** Put quotes around a phrase to search for it literally.
```
"how to reverse a linked list"
```

**Site-specific search.** Search within a specific website.
```
site:stackoverflow.com segmentation fault causes
```

**Exclude words.** Put a minus before a word to exclude it from results.
```
networking tutorial -beginner
```

**File type.** Search for a specific file format.
```
filetype:pdf operating systems notes
```

**In title or URL.** Narrow to pages where your term appears in the title or URL.
```
intitle:"getting started" compilers
inurl:blog system design
```

**Related sites.** Find sites similar to one you already know.
```
related:developer.mozilla.org
```

### Building Better Queries

The biggest improvement most people can make is being more specific. Instead of searching for how something works in general, search for the specific error, the specific behavior, or the specific thing you're trying to accomplish.

Compare:
- `why is my code slow` (too vague)
- `loop slow large list performance` (better)
- `list comprehension vs loop performance benchmark` (best)

If you get an error message, paste the exact message into the search with quotes around it. Remove anything specific to your machine or project path first.

When reading results, prefer primary sources. Official documentation over blog summaries. The actual paper over someone's interpretation of it. A library's own README over a third-party tutorial. Primary sources get updated. Secondary sources often don't.

### For Technical Questions

These search patterns tend to return useful results:

- `[concept] explained with examples`
- `[exact error message] solved`
- `[thing] vs [other thing] [year]` (include the year to get recent comparisons)
- `how does [concept] work internally`
- `[concept] common mistakes`
- `[tool] getting started guide`

### Using Documentation Search

Most good technical documentation has its own search. For anything you use regularly, learn to use the documentation's search instead of going through a general search engine. The official source is usually faster and more accurate for specific API or configuration questions.

Bookmark the documentation for everything you use. Return to it before searching elsewhere.
